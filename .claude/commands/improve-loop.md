## 全局规则（必须遵守）

1. **日志先行**：所有迭代记录写入 `logs/improve-loop.md`（不存在则创建）。每个"改进方向"完成后**立即**追加一条日志，然后才 commit + push。日志条目格式见下方模板。
2. **每个方向一次提交**：一个改进方向 = 一次代码改动 = 一次 `git commit` + `git push`。提交信息用中文，简明描述该方向。提交信息结尾保留：
   `Co-Authored-By: Claude Opus 4.8 <noreply@anthropic.com>`
3. **改前必测、改后必测**：任何代码改动后，必须跑验证（见"验证手段"），**绝不**在未验证的情况下声称完成或提交。验证失败先修复，修不好就回滚该方向并在日志记录原因。
4. **小步快跑**：一次只做一个方向，不要把多个无关改动塞进一次提交。
5. **不杜撰**：调研方向要基于真实读到的代码/运行结果，不要凭空编造。
6. **诚实收敛**：当你连续两轮都提不出有价值的新方向、且自测+真实用户测试全绿时，才宣布收敛并停止。

## 验证手段（按需组合）

- 后端冒烟（不花钱）：`python backend/selftest.py mock`
- 后端真实最短调用（极少额度，仅在涉及模型链路且确有必要时）：`python backend/selftest.py real`
- 前端类型+构建：在 `frontend/` 下 `npm run build`
- **真实用户测试**：启动应用（`scripts/dev.ps1` 或 `启动基金助手.bat`，端口 8766），像真实用户一样走通核心流程（critique / scheme / review / polish / 立项依据 / 合规），观察实际输出与交互，记录发现的问题。

## 循环协议

### 第 0 步：建立任务清单（仅首轮）
- 派一个agent，从小白用户角度触发，提出当前项目需改善的和缺少的功能

### 每一轮迭代：
对当前清单中的**每个子任务**，依次执行：

1. **调研改进方向**：阅读该子任务相关代码与实际运行表现，列出 0..N 个具体、可落地的改进方向（功能缺陷、健壮性、用户体验、性能、提示词质量、可维护性等）。若该子任务此轮无方向，跳过。
2. **挑选并改进**：从方向中选最高价值的一个，实施改动。
3. **验证**：跑相应验证手段（至少 mock 自测 + 受影响端的构建；涉及交互的方向要做真实用户测试）。失败则修复或回滚。
4. **记录日志**：向 `logs/improve-loop.md` 追加一条日志条目。
5. **提交**：`git add -A && git commit && git push`。
6. 回到该子任务步骤 1，直到该子任务本轮无新方向，再进入下一个子任务。

### 真实用户验收（每轮结束）
- 启动应用，以真实用户视角完整体验一遍，记录任何不顺手/报错/不符合预期之处，作为下一轮的新方向来源。

### 收敛判断
- 若**整轮**所有子任务都提不出新的有价值方向，且自测与真实用户测试均无遗留问题 → 进入"再确认轮"：再完整跑一轮调研。
- 若再确认轮仍无新方向、无遗留问题 → 在日志写"✅ 已收敛"总结（覆盖了哪些方向、当前已知限制），停止循环并向用户汇报。

## 日志条目模板（追加到 `logs/improve-loop.md`）

```
### [轮次 N · 子任务名] 方向标题
- 现状/问题：…
- 改进：…
- 验证：mock=✅/❌ build=✅/❌ 真实测试=…（具体观察）
- 提交：<commit 短 hash 或 "已 push">
```
